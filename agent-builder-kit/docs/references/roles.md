# 役割定義

## 基本方針
- `AGENTS.md` は全役割共通の原則と目次だけを持つ
- 詳細な責務、入力、出力、判断基準は各 docs に分離する
- 上流ほど抽象度が高く、下流ほど実装に近い成果物を扱う

## 標準 4 役モデル

### 1. プランオーナー
- 人間から受けた要求を `project-intake` に記録する
- 曖昧な要求を `discovery-brief` で整理し、仮置き前提と未確定事項を分ける
- `plan-spec` を確定し、task 化してよい境界を決める
- `program-board` を更新し、今どの plan を進めるかを決める

### 2. タスクプランナー
- `plan-spec` を chunk と ticket に分解する
- `editable_paths` と依存関係を切り分ける
- task 実行順と chunk 完了条件を定義する
- ワーカー完了後に進捗を集約する

### 3. タスクワーカー
- ticket に従って実装する
- 指定された検証を行う
- 完了時に事実のみを `fact-report` として返す
- review を通したうえで ticket 完了を申告する

### 4. reviewer
- task review と chunk review を行う
- findings を重要度順で返す
- バグ、回帰、境界逸脱、テスト不足、統合リスクを報告する

## 拡張 5 役モデル

上流の負荷が重いときだけ `プランオーナー` を 2 役へ分割する。

### 1. プランマネージャー
- 全体進捗の把握
- プランの優先順位付け
- 人間への確認依頼、提案、ガイド
- 次に仕様化すべきプランを `仕様設計者` へ渡す

### 2. 仕様設計者
- 人間との対話で曖昧な仕様を詰める
- 中規模仕様書を作る
- 制約、非目標、前提、未確定事項を整理する
- 仕様が固まった時点で `タスクプランナー` へ渡す

### 3. タスクプランナー
- 仕様書を chunk と ticket に分解する
- `editable_paths` と依存関係を切り分ける
- task 実行順と chunk 完了条件を定義する
- ワーカー完了後に進捗を集約する

### 4. タスクワーカー
- ticket に従って実装する
- 指定された検証を行う
- 完了時に事実のみを `fact-report` として返す
- review を通したうえで ticket 完了を申告する

### 5. reviewer
- task review と chunk review を行う
- findings を重要度順で返す
- バグ、回帰、境界逸脱、テスト不足、統合リスクを報告する

## なぜ標準を 4 役にするか
- `プランマネージャー` と `仕様設計者` はどちらも上流の意思決定で、責務軸が近いから
- 一方で `仕様設計者` と `タスクプランナー` は認知負荷の種類が違い、ここを統合すると再分解コストが増えやすいから
- 小中規模では `プランオーナー` が `project-intake` から `plan-spec` まで持つ方が流れが止まりにくいから

## どんなときに 5 役へ拡張するか
- 並行して進める plan が 2 本以上ある
- 人間確認が多く、上流で往復が多発する
- 非目標、制約、依存関係の整理が重い
- `タスクプランナー` が仕様待ちで止まりやすい
- 外部連携や複数サブシステムが絡み、仕様設計の滞留が全体進捗を止める

## 境界の切り方

### `プランオーナー` が決めること
- 今どの plan を進めるか
- 人間要求のどこを確認し、どこを仮置きするか
- task 化してよい状態まで仕様が固まったか
- どこで止めて人間判断へ戻すか

### `プランマネージャー` が決めること
- 今どのプランを進めるか
- どこで止めるか
- 人間判断が必要か

### `仕様設計者` が決めること
- 要求原文から何を仕様へ採用するか
- 何を作るか
- 何を作らないか
- どの制約下で作るか

### `タスクプランナー` が決めること
- どう分けて渡すか
- どこまでを 1 ticket にするか
- 並列化してよいか

### `タスクワーカー` が決めること
- ticket の範囲内でどう実装するか
- 最小の検証をどう通すか

### `reviewer` が決めること
- 差分に受け入れがたい問題があるか
- chunk として統合可能か

## role 切り替えの原則
- 標準は `プランオーナー` を使う
- `仕様設計者` を増やすのは、`タスクプランナー` を助けるためではなく、上流を分業した方が速いときだけ
- 単発タスクでも `project-intake` と `plan-spec` は省略せず、短く書く
- 単発タスクでは `chunk-sheet` を省略し、`plan-spec -> ticket` へ短縮してよい

## 役割間で受け渡す正本
- `Human -> プランオーナー`
  - 要求原文
- `プランオーナー -> タスクプランナー`
  - `project-intake`, `discovery-brief`, `plan-spec`
- `プランマネージャー -> 仕様設計者`
  - `program-board`
- `仕様設計者 -> タスクプランナー`
  - `project-intake`, `discovery-brief`, `plan-spec`
- `タスクプランナー -> タスクワーカー`
  - `chunk-sheet`, `ticket`
- `タスクワーカー -> 上流`
  - `fact-report`
- `reviewer -> 上流`
  - task review 結果、chunk review 結果
